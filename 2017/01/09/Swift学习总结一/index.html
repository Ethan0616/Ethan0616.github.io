lazy 延迟缓存：

必须将延迟存储属性声明成变量（使用var关键字），因为属性的初始值可能在实例构造完成之后才会得到。而常量属性在构造过程完成之前必须要有初始值，因此无法声明成延迟属性。
如果一个被标记为lazy的属性在没有初始化时就同时被多个线程访问，则无法保证该属性只会被初始化一次。

结构体和枚举是值类型。一般情况下，值类型的属性不能在它的实例方法中被修改。
并且它做的任何改变在方法结束时还会保留在原始的结构中。（注意：这里指变量）

d.构造类型：枚举、结构体和类

枚举类型只能包含实例计算属性，不能包含实例存储属性。

####### 结构体 与 类的区别
相同点：结构体有自己的方法，甚至拥有带存储方法的属性，几乎和类的功能一致。
不同点：
1.结构体不可以继承，类可以。
2.结构体是值传递，类为引用传递。
3.常量 let 定义的类实例，内部的的var变量属性是可以更改的。
4.常量let定义的结构体，无法修改该实例的任何属性，即使定义了var变量存储属性也不可以更改。

注：Swift不会真正的改变一个复制对象的值，直到它真的改变。
上面的话是重点，真正理解了这句，估计面试能加分。
什么意思呢？ 就是代码中的复制=号，并非立即再创建一个新对象，而是当原对象和新对象的值需要改变的时候，再进行创建。这是因为Swift管理所有的值拷贝以确保性能的最优化，只在绝对必要时才执行实际的拷贝。

常识：
Swift中，大多数基本类型，String、Array、Dictionary类型均以结构体的形式出现，值拷贝。Swift中结构体和枚举是值类型，类为引用类型。

Objective-C中，NSString、NSArray、NSDictionary均以类的形式，而非结构体，引用传递。

类的构造过程：
什么是类的构造？
设置实例每个存储属性的初始值和执行其他必须的设置或初始化工作

1.类和结构体在创建实例的时候，必须为所有存储属性设置合适的初始值。
2.构造器： 创建某个自定义实例的时候，的初始化方法。它的最简形式是不带任何参数的实例方法
init(){

}

3.默认属性值 ：
类或实例的属性的赋值时机有：1、声明属性的时候设置默认值 2、构造过程为属性设置初始值

4.自定义构造过程
可在构造器的方法中传入参数，对类或实例的属性进行赋值。构造过程可以修改常量属性的值。

5.构造参数 ： 自定义构造过程中传入的参数，语法和函数的参数相同

6.参数的内部名称和外部名称
构造函数和函数的方法参数一样 也需要一个供外部使用的名字，和一个在构造后，内部使用的名字。
如果在构造过程中没有传递参数，Swift会为构造器生成一个和内部参数名相同的外部参数名。

7.不带外部参数名的构造器
有的时候，不需要构造器内部有显示的参数名，那么用_ 来显示的描述外部参数名字，当外部调用构造器初始化的时候，就不需要带外部参数名称

8.可选属性类型
如果你定制的类型包含一个逻辑上允许取值为空的存储型属性——无论是因为它无法在初始化时赋值，还是因为它在之后某个时间点可以赋值为空——你都需要将它定义为可选类型optional type。可选类型的属性将自动初始化为nil，表示这个属性是有意在初始化时设置为空的。（——> 官方原话，我觉得很详细）
自己理解，可选值可以不在构造过程和初始化父类的过程中赋值。可以保留在类实例化后的任意时间去赋值。
原因，类的初始化过程要求，子类必须全部初始化自己的存储属性后，才能向上初始化父类。但有些值，只有在程序跑起来才能得真实的值。这时候，定义它为可选值，最贴切。可选值默认有值，值为nil。

9.构造过程中常量属性的更改
对于类的实例来说，它的常量属性只能在定义它的类的构造过程中修改；不能在子类中修改。
常量属性，只要在构造过程结束的时候，一旦有一个确定的值，它将永远不可更改。

10.默认构造器
如果类和结构体都有默认值，同时没有自定义构造器，那么Swift 会给这些结构体或类提供一个构造器，这个构造器将简单的创建一个所有属性都设置为默认值的实例。

11.结构体的逐一成员构造器 （同类的默认构造器相似，如果没有自定义构造方法，将自动生成一个，名称不同而已，叫做逐一成员构造器）

  1. 值类型的构造器代理
    构造器相互调用就是 构造器代理
    构造器代理实现的规则在值类型和类类型中略有不同。
    1、值类型（结构体和枚举类型）不支持继承，只能代理给提供它的构造器。 值类型，直接调用self.init 在自定义的构造器中引用类型中的其他构造器。只能在构造器内部调用self.init。
    如果值类型已经有一个自定义构造器，那么将无法访问到默认构造器（如果是结构体，还将无法访问逐一成员构造器）
    这么做的原因： 防止已经有了其他构造器，他人调用的时候，还是调用默认构造器。
    假如你希望默认构造器、逐一成员构造器以及你自己的自定义构造器都能用来创建实例，可以将自定义的构造器写到扩展（extension）中，而不是写在值类型的原始定义中。想查看更多内容，请查看扩展章节。

2、类则不同，可继承自其他类，类有责任保证其所有继承的存储类型属性在构造时也能正确的初始化。

13、类的继承和构造过程
类内所有的存储属性，包括父类的属性 ，必须在构造过程中设置初值。
Swift 为类提供了两种构造器
1.指定构造器
必须总是向上代理 Designated
主要的构造器，每一个类都必须调有至少一个指定构造器。
init(parameters) {
statements
}
规则： 指定构造器必须调用其直接父类的指定构造器

2.便利构造器
必须总是横向代理 Convenience
辅助型构造器，可以定义便利构造器来调用同一个类中的指定构造器，并为其参数提供默认值。
convenience init(parameters) {
statements
}

规则： 便利构造器必须调用同一类中定义的其他构造器
规则2 ： 便利构造器必须最终导致一个指定构造器被调用

14.两段式构造过程
第一个阶段，每个存储型属性通过引入它们的类的构造器来设置初始值
第二阶段，开始前，当每一个存储型属性值被确定后，它给每个类一次机会在新实例准备使用之前进一步定制它们的存储型属性

1.必须先初始化自己的所有属性，才能向上代理。
2.如果有继承，在父类初始化结束后再为继承的属性设置新值（不这么做，将被父类的值覆盖）
3.便利构造器，必须先调用同类中其他的构造器，然后再为任意属性赋予新值（如不这么做，设置好的初值将被其他构造器所覆盖）
4.构造器在第一阶段构造完成之前，不能调用任何实例方法，不能读取任何实例属性的值，不能引用self作为一个值。

15.构造器的继承和重写
1.子类的指定构造器 or 便利构造器，当与父类指定构造器名称和参数相同，实际上是在重写父类的指定构造器，需要加上关键字override
class Food {
var name: String
init(name: String) {
self.name = name
}
convenience init() {
self.init(name: “[Unnamed]”)
}
}
class RecipeIngredient: Food {
var quantity: Int
init(name: String, quantity: Int) {
self.quantity = quantity
super.init(name: name)
}
override convenience init(name: String) {
self.init(name: name, quantity: 1)
}
}

2.子类的便利构造器，当与父类便利构造器名称和参数相同，实际上只是子类的便利构造器，与父类没有毛关系，因此不需要带上override关键字。
override关键字的作用，让编辑器去检查父类中是否有相匹配的指定构造器，并且验证构造器参数是否正确。

注：子类可以在初始化时修改继承来的变量属性，但是不能修改继承来的常量属性。

16.构造器的自动继承
前提：子类新引入的所有属性都有初值
1.如果子类没有定义任何指定构造器，它将自动继承所有父类的指定构造器
2.如果子类提供了所有父类指定构造器的实现，无论是自动继承，还是子类自己提供的实现，只要是全部继承了父类的指定构造器，那么也会自动继承父类的便利构造器。
当子类构造器实现了父类全部指定构造器，子类会自动继承父类所有的便利构造器。（这样做的原因，我们试想，假设没有继承父类的指定构造器A，当调用父类某个便利构造器，而便利构造器调用了这个A，可能会有存储属性初始化值遗漏现象。所以子类必须实现父类所有指定构造器，这时，子类自动继承了父类的便利构造器）

17.可失败构造器
当构造过程，参数不满足我们的需求，或者外部资源不足的情况下，需要初始化失败。这时，可以在一个类、结构体、或者枚举类型的定义中，添加一个可失败构造器。语法 init?()，注意可失败构造器的参数名和参数类型，不能与其它非可失败构造器的参数名，及其参数类型相同。return nil

1.种类有枚举类型的可失败构造器，带原始值的枚举类型的可失败构造器，
3.值类型的可失败构造器，可以在构造过程中任意时间点触发失败（甚至在属性赋值前，构造一开始就return nil）
4.类只能在类引入的所有存储属性被初始化以后，并且构造器代理调用完成以后，才能触发构造失败。
class Product {
let name: String!
init?(name: String) {
self.name = name
if name.isEmpty { return nil }
}
}

构造失败的传递
子类非可失败构造器中调用父类可失败构造器，需要强制解包
class UntitledDocument: Document {
override init() {
super.init(name: “[Untitled]”)!
}
}
可失败构造器 init!
该可失败构造器会构建一个对应类型的隐式解包可选类型的对象。

必要构造器 在类的构造器前面加required 关键字，表示所有该类的子类都必须实现该构造器
在子类重写父类的必要构造器的时候，子类的构造器前也要加required关键字，表明该构造器，也要应用于后面的子类。
在重写父类中必要的指定构造器时，不需要添加override
如果子类中继承的构造器能够满足必要的构造要求，则不需要显示的提供必要构造器的实现。

通过闭包或函数设置属性的默认值
必须立刻调用closure，利用返回值，给属性赋值。如不用返回值，只是给closure指定一个闭包类型。而未对其赋初始值。
注意：当使用closure来初始化属性的时候，部分实例还没有初始化。所以不能访问其他属性，也不能用self属性，或者调用任何实例方法。

e. 协议

协议也是一种类型
1.协议作为函数、方法或构造器中的参数类型或返回值类型
2.作为常亮、变量或属性的类型
协议：
不允许结构体或枚举等值类型集成协议
需要在protocol 后面加上 class

在Swift中声明一个协议
1.通常协议用关键字var 来声明属性，
2.在类型声明后面加上{set get}表示是否为可读可写属性
3.总是使用static作为前缀

Equatable协议

####### Static 与Class 区别
在 Swift 1.2之前，protocol中使用的是Class关键字，但不合理；1.2后，除了确实是在具体的class中实现以外，其他情况下都使用static。

class 用来修饰类类型 可继承
static 用来修饰值类型 不可继承
Swift 不允许在类中定义class修饰的类型存储属性，只能定义static修饰的类型存储属性。

实际问题解决

解决实例之间的循环引用： 
1.弱引用（weak reference）：对声明周期中可能会变成nil的实例使用弱引用 
2.属性声明前 加上  weak 关键字，弱引用必须声明为变量。
3.将每一个弱引用声明为可选类型，因为弱引用没有值。
weak var tenant: Person?

无主引用（unowned reference）：
1.初始化赋值后再也不会被赋值为nil的实例，使用无主引用。
2.无主引用永远有值
3.无主引用要定义为非可选类型
4.声明属性或者变量时，需要加上关键字 unowned 
5.无主引用可以直接被访问，不需要解包。不能将无主引用赋值为nil，因为非可选类型变量不允许赋值为nil

闭包引起的循环引用问题
在定义闭包时同时定义捕获列表作为闭包的一部分，通过这种方式可以解决闭包和类实例之间的循环强引用。捕获列表定义了闭包体内捕获一个或者多个引用类型的规则。跟解决两个类实例间的循环强引用一样，声明每个捕获的引用为弱引用或无主引用，而不是强引用。应当根据代码关系来决定使用弱引用还是无主引用。
Swift中闭包调用实例的属性时，需要显示的用self  因为怕用户不小心捕获了self，造成循环引用的问题。
定义捕获列表
lazy var someClosure: Void -> String = {
[unowned self, weak delegate = self.delegate!] in
// 这里是闭包的函数体
}


闭包中的弱引用和无主引用
在闭包和捕获的实例总是相互引用并且总是同时销毁，将闭包内的捕获定义为无主引用。
在被捕获的引用可能变为nil时，将闭包内的捕获定义为弱引用。
如果被捕获的值永远不能为nil ，那么应该使用无主引用。
